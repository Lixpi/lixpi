# Lixpi — Tauri 2 native app

This package wraps the existing Svelte SPA at `services/web-ui/` as a native macOS desktop and iOS mobile app using Tauri 2.

## One-time host setup

1. **Install Rust** (required for the Rust backend Tauri compiles into your native app):
   ```sh
   curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
   source "$HOME/.cargo/env"
   rustup target add aarch64-apple-darwin x86_64-apple-darwin
   ```

2. **Install Xcode + Command Line Tools** for macOS and iOS builds:
   ```sh
   xcode-select --install
   ```

3. **Install Node deps** from the workspace root:
   ```sh
   pnpm install
   ```

4. **Generate platform icons** from the existing logo (one-time):
   ```sh
   cd gui/tauri
   pnpm icon
   ```
   This populates `src-tauri/icons/` from `services/web-ui/public/logo-512.png`.

5. **Add Tauri WebView origins to NATS allowed_origins** in your dev/staging env so the WebSocket handshake is accepted. In your `.env.<stage>` file, append to `NATS_ALLOWED_ORIGINS`:
   ```
   tauri://localhost,http://tauri.localhost
   ```
   (The exact origin string for iOS WKWebView should be confirmed during the spike by inspecting the WSS request in the WebKit Inspector.)

6. **Auth0 dashboard** — add to **Allowed Callback URLs** and **Allowed Logout URLs**:
   - `lixpi://auth/callback` (macOS desktop)
   - `https://app.lixpi.com/auth/callback` (iOS Universal Link — same as web)

   For iOS, also host the `apple-app-site-association` JSON at `https://app.lixpi.com/.well-known/apple-app-site-association` referencing bundle ID `com.lixpi.app` and path `/auth/callback`.

7. **Build env** — use a per-target `VITE_AUTH0_REDIRECT_URI`:
   - web build → `https://app.lixpi.com/auth/callback`
   - macOS Tauri build → `lixpi://auth/callback`
   - iOS Tauri build → `https://app.lixpi.com/auth/callback`

## Daily commands

```sh
# macOS desktop dev (hot-reload Vite + auto-restart Tauri on Rust changes)
cd gui/tauri && pnpm dev

# macOS production build (signed/notarized .dmg lands in src-tauri/target/release/bundle/)
cd gui/tauri && pnpm build

# iOS — one-time scaffold of the Xcode project
cd gui/tauri && pnpm ios:init

# iOS dev on a physical device or simulator (Vite must be reachable from device LAN)
cd gui/tauri && pnpm ios:dev

# iOS archive build for TestFlight
cd gui/tauri && pnpm ios:build
```

The Tauri build references `../../../services/web-ui/dist` as `frontendDist`, so the SPA at `services/web-ui/` builds via `pnpm --filter lixpi-web-ui build` (called automatically by `tauri build`).

## What changed in the SPA

Three small patches landed in `services/web-ui/`:

- `src/services/auth0-service.ts` — when running under Tauri, subscribes to the deep-link plugin and routes the OAuth callback URL into `auth0.handleRedirectCallback(url)`. The web build is unaffected (the dynamic import is gated on `__TAURI_INTERNALS__`).
- `src/main.ts` — at startup, if running under Tauri, reads an optional `nats_server_url` override from the Tauri store before initializing NATS. Falls back to `VITE_NATS_SERVER` otherwise.
- `index.html` — viewport meta now includes `viewport-fit=cover` for iOS safe-area support.
- `package.json` — adds `@tauri-apps/api`, `@tauri-apps/plugin-deep-link`, `@tauri-apps/plugin-store` so Vite can resolve the dynamic imports at build time.

## Backend connection

By default the app connects to the production cloud NATS endpoint (the `VITE_NATS_SERVER` baked into the build). The Settings panel (to be added in Phase 1) lets desktop power users override this URL at runtime; the override is persisted via Tauri's plugin-store and read on next startup.

Mobile builds will not expose the override UI — iOS App Store review tends to flag user-configurable backend URLs.

## Notes & caveats

- The Tauri build host MUST be macOS for the macOS and iOS targets — Apple licensing.
- Custom CA certs from local `init-infrastructure.sh` will not be trusted on iOS; mobile dev must point at a cloud-dev NATS endpoint with a real cert.
- The Stripe.js `<script>` in `index.html` is allowed by the CSP but no purchase UI ships in v1 (defers App Store digital-goods compliance).
- Tauri 2 mobile is production-usable but plugin coverage is thinner than React Native — budget contingency for any missing native API we hit (e.g., specific iOS file picker behavior).
